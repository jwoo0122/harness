# Iteration 9 Criteria — Managed Worktree Core

## Goal
Establish the first real worktree-management foundation for harness so that:

1. harness can create a **managed git worktree** for a new independent task
2. the current pi runtime can switch into a **session rooted at that worktree**
3. managed worktrees are tracked in **shared repo metadata**, not per-worktree cwd-local state
4. disposable managed worktrees can be **cleaned up safely**
5. harness-controlled mutation can be **blocked unless the current session has a valid managed-worktree binding**

This iteration intentionally ships the **core manager + managed-workspace lifecycle + non-user-facing orchestration entrypoint** first.
It does **not** yet make built-in `/new` automatically create worktrees by default.

## Acceptance Criteria

### AC-9.1 — Managed worktree identity is explicit and repo-shared
Create a dedicated worktree-management seam/module for harness-managed worktrees.

Expected shape:
- harness-managed worktrees use an explicit unique ID
- harness-managed branches use a deterministic prefix such as `harness/wt/<id>`
- shared worktree metadata/lease files are stored under the repo’s **git common dir**
  - e.g. via `git rev-parse --git-common-dir`
- no worktree-management registry/lease state for this feature is stored under cwd-local `.harness/`
- the manager must distinguish **harness-managed** worktrees from user-created worktrees

### AC-9.2 — Internal managed-worktree creation entrypoint creates a managed worktree and switches into it
Add a **non-user-facing** harness-internal entrypoint for managed worktree creation.

Allowed implementation shapes:
- internal helper/module API
- tool → internal command handoff
- equivalent internal orchestration path
- reserved internal extension command only if it can remain out of normal user-facing slash-command discovery

User-facing constraint:
- an explicit creation entrypoint may exist in code, but users must not be expected to discover or manually invoke it as part of the normal workflow
- harness docs/skills/status/help text for this iteration must not present managed worktree creation as a public slash-command workflow
- if pi cannot practically hide an extension command from normal user-facing command discovery, this iteration should prefer a helper/tool-driven internal path instead of a user-visible slash command

Expected behavior:
- the internal entrypoint is designed to be invoked by harness orchestration/tests when a managed workspace is required; users are not expected to invoke it directly
- the internal entrypoint fails clearly outside a git repo
- it allocates a unique managed worktree ID, branch name, and worktree path
- worktree creation uses an explicit branch-creating flow such as:
  - `git worktree add -b <branch> <path> HEAD`
- the target worktree root lives under a deterministic managed location
  - e.g. sibling `.worktrees/<repo>/<id>` or equivalent
- if the current cwd is a subdirectory below the repo root, the target session cwd preserves that relative subdirectory when possible
- after creation, harness switches the current pi runtime into a session rooted at the target cwd
- built-in `/new` behavior remains unchanged in this iteration

### AC-9.3 — Dirty-state preflight prevents silent loss of local edits
The managed-worktree creation flow must not silently imply that current uncommitted changes were copied to the new worktree.

Expected behavior:
- if the current checkout is dirty, the flow must either:
  - cancel, or
  - require explicit confirmation that the new worktree is based on `HEAD` only
- user-facing messaging must state that uncommitted changes are **not** automatically carried into the new worktree
- this iteration must **not** auto-stash, auto-apply, or auto-cherry-pick local edits

### AC-9.4 — Target-cwd session creation is durable and cwd-correct
The worktree flow must create a **persisted** session file for the target cwd before switching into it.

Expected behavior:
- the implementation must not rely on `ctx.newSession()` alone for cross-cwd switching
- `ctx.switchSession(sessionFile)` must receive a real session file path rooted at the target cwd
- if a small shim/manual session-file creation step is needed, that is acceptable
- after switching:
  - `ctx.cwd` reflects the target cwd
  - `ctx.sessionManager.getCwd()` reflects the target cwd
  - shell/process cwd checks align with the target cwd
- if pi `sessionDir` is configured as a **relative** path, the target session path must resolve relative to the **target worktree root**, not the source cwd

### AC-9.5 — Session ↔ worktree association survives restart and resume
A managed session must remain associated with its managed worktree over time.

Expected behavior:
- resuming a session created via the managed-worktree creation flow returns to the same managed worktree cwd
- if the managed worktree path is missing or moved, the failure must be explicit
  - no silent fallback into the wrong cwd
- harness status/UI must surface the current managed worktree identity when applicable
  - at minimum: ID and/or branch
  - path visibility is desirable if it fits the existing status surface

### AC-9.6 — Managed-workspace lifecycle state is explicit and deterministic
Managed worktrees must be organized by an explicit managed-workspace lifecycle, not only by ad hoc helper calls.

Expected shape:
- each harness-managed worktree has one shared lease record in the git-common-dir-backed registry
- lease data includes enough information to reconcile session/worktree state, including at least:
  - worktree ID
  - worktree path
  - branch
  - base commit
  - session file
  - last-seen timestamp
  - lease expiration
  - lifecycle state
- the implementation uses explicit lifecycle states for managed workspaces, such as:
  - unmanaged
  - provisioning
  - managed-active
  - managed-released
  - managed-missing
  - manual-cleanup-required
  - or an equivalent small deterministic state model
- persisted lease state must stay minimal and auditable; facts that can be re-derived from Git/worktree inspection should not be blindly trusted from stale stored metadata

Expected lifecycle behavior:
- renew/activate on `session_start`
- refresh during active usage (e.g. `turn_end`)
- release softly on `session_shutdown`
- transition to an explicit missing/error state if the bound managed worktree path no longer exists
- only harness-created managed worktrees receive or update these leases/states

### AC-9.7 — Authoritative mutation gate blocks harness-controlled mutation outside a valid managed binding
Harness must not rely on convention alone to keep independent work inside managed worktrees.

Expected behavior:
- there is one authoritative gate/check path for harness-controlled mutation decisions
- at minimum, the gate covers harness-controlled mutation paths such as:
  - managed-worktree creation/switch bootstrap
  - janitor removal
  - execute-path `write`
  - execute-path `edit`
  - execute-path mutating `bash`
  - `harness_commit`
  - or an explicitly documented equivalent set of harness-controlled mutation paths
- if the current session is expected to operate inside a managed workspace, mutation must be blocked unless:
  - there is exactly one valid managed-worktree binding for the session
  - `ctx.cwd` is inside that managed worktree root/target cwd
  - lifecycle state and live Git facts allow the operation
- blocked mutation must fail with an explicit, debuggable reason
- the guarantee claim for this iteration is limited to **harness-controlled mutation paths**
  - this iteration does **not** claim full OS/kernel sandboxing or protection from arbitrary out-of-band mutation

### AC-9.8 — Inline janitor safely prunes disposable managed worktrees
Add a safe janitor path for managed worktrees.

Expected behavior:
- janitor runs at safe lifecycle boundaries such as:
  - before invoking the internal managed-worktree creation flow
  - on startup/session start
- optional explicit manual command is allowed, but not required for this AC
- janitor uses Git worktree state as a source of truth
  - e.g. `git worktree list --porcelain`
- janitor may auto-remove a managed worktree **only if all are true**:
  - it is harness-managed
  - its lease is expired
  - the worktree still exists
  - the worktree is clean
  - it has no unique commits beyond its recorded base
- dirty/diverged expired managed worktrees must **not** be auto-removed
  - they must be retained and surfaced as manual-cleanup-required
- janitor must never mutate or remove non-harness-managed worktrees

### AC-9.9 — Validation covers worktree manager behavior deterministically
`package.json` `validate:extensions` must cover the new worktree-management seam in a way that makes VER’s job straightforward.

Validation must make it possible to confirm:
- a non-user-facing managed-worktree creation entrypoint exists and is wired for orchestration/tests
- harness docs/skills/help/status do not present that entrypoint as a public user workflow
- if the implementation uses a command-shaped internal entrypoint, it is not relied on as a normal user-discoverable slash-command workflow
- managed metadata resolves under the git common dir, not `.harness/`
- cross-cwd session switching lands in the target worktree cwd
- relative `sessionDir` handling is correct for the target worktree
- restart/resume preserves the session↔worktree association
- lifecycle-state transitions are deterministic for the covered scenarios
- harness-controlled mutation is blocked outside a valid managed binding
- janitor never removes non-managed worktrees
- janitor removes only clean expired disposable managed worktrees
- built-in `/new` remains unchanged in this iteration

A temp-repo smoke test is acceptable and encouraged for this iteration.

## Explicit Non-Goals
- no default automatic `/new` → worktree reroute yet
- no public documented `/worktree-new`-style user workflow in this iteration
- no natural-language task-boundary detection yet
- no background daemon, file watcher, or always-on janitor service
- no automatic migration of uncommitted local edits into the new worktree
- no Local↔Worktree handoff/merge UX yet
- no automatic management of user-created non-harness worktrees
- no claim of full OS/kernel sandbox isolation in this iteration
- no verification-registry schema/path redesign
- no execute-mode protocol redesign beyond using the new worktree manager foundations later
