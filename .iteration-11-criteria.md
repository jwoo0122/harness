# Iteration 11 Criteria — Execute Auto-Bootstrap into Managed Worktrees

## Goal
Build on Iteration 9's managed-worktree core so that `/execute` no longer depends on a manually invoked internal bootstrap step.

This iteration should make **execute-mode task startup automatically acquire or reuse the correct managed worktree** while keeping worktree creation **non-user-facing**.

Specifically, this iteration should ensure that:

1. starting `/execute` from an unmanaged checkout can automatically route into a clean managed worktree before implementation begins
2. starting `/execute` from a valid managed task session reuses that managed workspace instead of creating duplicates
3. users are still **not** expected to discover or invoke a worktree command directly
4. dirty-source behavior remains explicit and safe

This iteration intentionally targets the **`/execute` orchestration boundary**.
It still does **not** make built-in `/new` auto-reroute by default.

## Acceptance Criteria

### AC-11.1 — `/execute` auto-bootstrap uses managed worktrees when needed
When an `/execute` run starts from a session/cwd that is not already bound to a valid managed worktree, harness must automatically route the run through the managed-worktree bootstrap path before execute-role implementation begins.

Expected behavior:
- the parent `/execute` orchestration path detects whether the current session already has a valid managed-worktree binding
- if not, harness automatically invokes the internal managed-worktree bootstrap path
- the user does not need to know or manually invoke any worktree-specific command/tool name
- after bootstrap/switch completes, the execute run continues in the target managed session/worktree
- the resulting execute subagents inherit the managed binding and target cwd

### AC-11.2 — Valid managed execute sessions are reused, not duplicated
If `/execute` starts from a session that is already correctly bound to an active managed worktree, harness must reuse it instead of allocating another managed workspace.

Expected behavior:
- valid existing binding -> continue in place
- no duplicate worktree creation for the same active execute task/session
- if the binding is stale, missing, or invalid, the failure must be explicit or the bootstrap path must repair/recreate deterministically
- status/output should make it clear whether execute reused an existing managed workspace or allocated a new one

### AC-11.3 — Dirty unmanaged source checkout blocks or requires explicit execute-time confirmation
Auto-bootstrap at the `/execute` boundary must preserve the dirty-state safety guarantee from Iteration 9.

Expected behavior:
- if `/execute` is started from an unmanaged dirty checkout, execute must not silently continue as if local edits moved into the managed worktree
- harness must either:
  - stop and ask for explicit confirmation that execute should proceed from `HEAD` only, or
  - abort the execute startup cleanly
- user-facing messaging must still state that uncommitted changes are not automatically carried over
- this iteration still must not auto-stash, auto-apply, or auto-cherry-pick local edits

### AC-11.4 — Execute startup sequencing is deterministic across the session switch boundary
The `/execute` protocol must survive the auto-bootstrap session switch without losing criteria context or accidentally running mixed work before/after the switch.

Expected behavior:
- criteria/objective context survives the bootstrap/switch boundary
- execute-role planning/implementation does not begin in the wrong cwd before the switch completes
- after the switch, the resumed execute run uses the target managed session/worktree cwd consistently
- duplicate parent execute runs are avoided
- if bootstrap fails, execute should fail clearly instead of partially continuing in the source checkout

### AC-11.5 — Public workflow remains worktree-opaque
This iteration must improve automation without surfacing worktree management as a user workflow.

Expected behavior:
- docs/help/skills must not tell users to invoke an internal worktree bootstrap command as part of normal usage
- `/execute` should be the visible user entrypoint for this behavior
- if any internal command/tool still exists in code, it remains an implementation detail rather than a documented public workflow

### AC-11.6 — Validation deterministically covers execute auto-bootstrap behavior
`package.json` `validate:extensions` must cover the new execute auto-bootstrap behavior deterministically.

Validation must make it possible to confirm:
- `/execute` detects unmanaged vs valid-managed starting state
- unmanaged execute startup routes through the managed bootstrap path
- already-managed execute startup reuses the existing binding
- dirty unmanaged startup blocks or requires explicit HEAD-only confirmation
- execute context survives the session switch boundary correctly
- public docs/skills still do not present a user-facing worktree command workflow
- built-in `/new` remains unchanged in this iteration

A temp-repo/runtime smoke strategy is acceptable and encouraged.

## Explicit Non-Goals
- no default built-in `/new` auto-reroute yet
- no natural-language task-boundary detection yet
- no automatic migration of local uncommitted edits
- no broader auto-worktree behavior for non-execute prompts yet
- no daemonized background task/worktree scheduler yet
