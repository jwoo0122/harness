# Iteration 10 Criteria — Verification Accumulation Core

## Goal
Redefine verification accumulation around the single most important concern:

> **cumulative accumulation of reproducible verification methods with authoritative executed evidence**

This iteration must stay tightly scoped to that core. It upgrades the verification registry from a mutable summary of pass/fail into a split system:
- committed reusable verification specs
- runtime immutable verification receipts

## Acceptance Criteria

### AC-10.1 — Committed registry becomes a reusable spec catalog
`.harness/verification-registry.json` must become the committed authority for reusable verification specs, not mutable pass/fail state.

Required properties:
- schema upgrades to `harness-verification-registry-v2`
- top-level shape uses `specs`, keyed by stable `check_id`
- each spec stores reusable verification definition data
- each spec can bind one or more AC/requirement bindings

The committed registry must not require runtime receipt data to remain committed in Git.

### AC-10.2 — Legacy v1 data migrates without importing old pass/fail truth
Existing v1 registry data must load successfully and be migrated into the v2 spec model.

Migration rules:
- legacy entries become reusable specs
- legacy `lastResult` and `lastVerifiedAt` must **not** become authoritative runtime truth
- writes must emit only v2 schema

### AC-10.3 — Runtime receipts are repo-shared and append-only
Executed verification evidence must live outside the committed registry under the repository's git common dir:

```text
<GIT_COMMON_DIR>/pi-harness/verification/
```

Required properties:
- linked worktrees for the same repository share the same runtime receipt store
- receipt history is append-only
- reruns append new receipts instead of overwriting prior evidence

### AC-10.4 — Runtime corruption fails explicitly
Malformed runtime receipt state must not silently degrade to empty success-like behavior.

Required properties:
- missing runtime receipt store may initialize as empty
- malformed runtime receipt content must raise an explicit error
- silent fallback on runtime corruption is forbidden

### AC-10.5 — Registering a spec must not claim a pass
`harness_verify_register` must register or update reusable verification specs only.

Required properties:
- register/update must not stamp authoritative pass/fail status
- register/update must not write `lastResult` / `lastVerifiedAt` as truth
- repeated registration should update the reusable spec while preserving accumulated bindings history where appropriate

### AC-10.6 — Executed evidence gets an authoritative tool surface
A verification execution path must exist for automated specs.

Required properties:
- add `harness_verify_run`
- automated specs execute their registered command
- every execution appends an immutable receipt
- receipt data records enough context to derive freshness and latest outcome

### AC-10.7 — Listing/status becomes receipt-driven and freshness-aware
`harness_verify_list` must derive latest status from receipts, not mutable repo summary fields.

Required statuses:
- `pass`
- `fail`
- `missing`
- `stale`

Freshness must consider at least:
- current spec identity/version
- current repository HEAD

### AC-10.8 — Deterministic validation covers migration, sharing, and corruption behavior
`npm run validate:extensions` must deterministically validate the new verification accumulation model.

Validation must cover:
- v1 → v2 migration
- committed registry round-trip in v2 shape
- append-only runtime receipts
- explicit failure on malformed runtime state
- receipt-derived pass/fail/missing/stale status
- linked-worktree sharing through git common dir
- `extensions/index.ts` wiring for `harness_verify_register`, `harness_verify_run`, and `harness_verify_list`

## Explicit Non-Goals
This iteration does **not** require:
- full execute-protocol redesign beyond verification accumulation
- full mutation-gate enforcement across every path
- sophisticated affected-check scheduling
- retroactive trust of legacy `lastResult` / `lastVerifiedAt`
- converting every historical manual check into automated execution
