# Iteration 7 Criteria ‚Äî Extension Refactor Slice 3

## Goal
Continue the `extensions/index.ts` decomposition using the agreed sequence:

1. Plan
2. Validate (prepare)
3. Implement
4. Validate

This iteration covers only the **bash-policy classification seam**. It intentionally does **not** extract tool handlers, web helpers, lifecycle orchestration, `extensions/subagents.ts`, or `extensions/compact-tool-renderers.ts`.

## Acceptance Criteria

### AC-7.1 ‚Äî Prepare entrypoint is repointed to the bash-policy seam
Before bash-policy extraction, `package.json` must wire:
- `validate:extensions:prepare` ‚Üí bash-policy-specific prepare validation

The registry-seam prepare hook must remain in the repo, but it must no longer be the active prepare entrypoint.

### AC-7.2 ‚Äî Bash-policy prepare gate works without peer dependency installs
`npm run validate:extensions:prepare` must pass on this repo without requiring local installs of:
- `@mariozechner/pi-coding-agent`
- `@mariozechner/pi-tui`
- `@sinclair/typebox`

The prepare gate must verify by text/FS inspection that:
- the bash-policy seam still exists in `extensions/index.ts`
- extraction has not already happened yet
- adjacent non-scope helpers still remain in `extensions/index.ts`
- prior prepare validators still exist in the repo

### AC-7.3 ‚Äî Bash-policy seam is extracted into a dedicated module
Create `extensions/bash-policy.ts` and move the bash-policy seam out of `extensions/index.ts`, including:
- `READ_ONLY_BASH_PREFIXES`
- `MUTATING_BASH_PREFIXES`
- `RAW_NETWORK_BASH_PREFIXES`
- `VERIFY_BASH_PREFIXES`
- `isAgentBrowserCommand`
- `classifyExploreBash`
- `classifyExecuteBash`
- `classifyChildBashCommand`

### AC-7.4 ‚Äî Extracted bash-policy module is leaf-like
`extensions/bash-policy.ts` must:
- not import from `extensions/index.ts`
- not import `@mariozechner/*`
- not import `@sinclair/typebox`
- preserve current `startsWith` matching and branch ordering semantics

### AC-7.5 ‚Äî Root wiring is updated without scope expansion
`extensions/index.ts` must import bash helpers from `./bash-policy.js` and must no longer define the extracted bash-policy seam.

The following must remain in `extensions/index.ts` for this increment:
- `parseHarnessSubagentBashPolicy`
- all tool handlers
- web helper functions
- lifecycle/event wiring

### AC-7.6 ‚Äî Post-implementation validator proves bash-policy behavior parity
`npm run validate:extensions` must pass and verify:
- the existing prompt validation still runs
- the existing registry validation still runs
- the bash-policy module can be syntax-checked
- source-shape checks confirm the extraction is real
- representative explore-mode classification behavior is preserved
- representative execute-mode classification behavior is preserved
- child-policy delegation behavior is preserved
- at least one overlapping prefix/order sentinel is preserved

### AC-7.7 ‚Äî Validation chain remains forward-compatible
The full validation chain must not hardcode ownership of `validate:extensions:prepare` to the prior registry-seam prepare script.

After this increment:
- `validate:extensions` must remain green
- prior full validators must still verify their seam without blocking future prepare-entrypoint rotation

### AC-7.8 ‚Äî Scope does not expand into bash-policy redesign
No changes in this iteration may:
- redesign bash-policy semantics
- add shell parsing/tokenization
- change tool names or tool schemas
- broaden `agent-browser` handling beyond preserving current behavior
- fix unrelated command-allowlist quirks

## Explicit Non-Goals
- no extraction of web helpers yet
- no extraction of tool registration/handlers yet
- no refactor of `extensions/subagents.ts`
- no refactor of `extensions/compact-tool-renderers.ts`
- no `agents/` layout changes
- no registry path/schema changes
